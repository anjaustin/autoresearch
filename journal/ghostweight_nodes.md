# Nodes: Project GhostWeight Tradeoffs

## Decision Points

### Node 1: PRNG Initialization Strategy
**Question:** How do we re-seed for each row/block to ensure determinism?

Options:
- A) **Full State Recompute:** Re-hash (seed, layer_id, row) for every row using a strong hash like SpookyHash or MurmurHash.
- B) **State Offset (Linear):** Initialize once per layer, then skip the PRNG forward. (Fast, but Xorshift is not easily jumpable).
- C) **Deterministic Row Seed:** Use a simple mixing function `seed ^ layer_id ^ row_index` to initialize 4 lanes of Xorshift state at the start of each row.

Tension: Option C is fastest but might introduce row-to-row correlation if the mixing function is weak.

### Node 2: Weight Generation Granularity
**Question:** Do we generate 8 weights at a time, or generate a large buffer per row?

Options:
- A) **JIT Generation:** Generate 128 weights (one 256-bit PRNG draw), then consume them in 8-weight AVX2 chunks.
- B) **Buffered Generation:** Generate all K weights (e.g., 2560) into an L1-cache buffer at the start of each row.

Tension: JIT (Option A) keeps everything in registers (fastest). Buffered (Option B) allows reusing weights for multiple tokens in a batch (M > 1), but takes up L1 space.

### Node 3: The Bit-Mapping Efficiency
**Question:** Can we map PRNG bits to {-1, 0, 1} without branch divergence?

Mapping Choice:
- Bit 0: Non-zero mask (0=zero, 1=non-zero)
- Bit 1: Sign (0=pos, 1=neg)
- To get 50/25/25 distribution:
    - Bit 0 must be 1 only 50% of the time.
    - Bit 1 is 50% pos/neg.

Logic:
`nz_mask = (bits & 1)`
`sign_mask = (bits & 2)`
Then use the same XOR/AND trick from our v3 kernel.

### Node 4: Backward Pass Re-generation
**Question:** How to handle `grad_input = W^T @ grad_output` efficiently?

Since `W` is generated by rows, we must either:
1. Generate row `i`, then immediately apply its contribution to all output elements (requires scatter-add).
2. Transpose the PRNG (impossible).
3. Generate `W` once and store it in RAM for the backward pass.

Decision: We will use the **Generate-and-Scatter** approach. It maintains the zero-storage requirement.

### Node 5: LM-Head Synthesis
**Question:** How to compress the 320MB LM-head?

Options:
- A) Keep it as-is (FP32 patch). The model is now 320MB instead of 800MB.
- B) Procedural Synthesis using a **Low-Rank Random Approximation**. 
- C) TinyLoRA for LM-Head: `LM_Head(x) = PRNG_Matrix(x) + A(B(x))`.

Constraint: For Phase 1, we focus on the Decoder. We will keep the LM-Head as-is until the decoder Ghosting is validated.
